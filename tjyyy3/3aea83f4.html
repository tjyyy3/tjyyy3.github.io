【Git】如何参与开源项目

发表于2025-03-10|更新于2026-02-10
|字数总计:1.8k|阅读时长:6分钟|阅读量:

记录一下参与开源项目以及做贡献（提交PR）的过程和步骤。

一、选择合适的开源项目

作为新手，选择合适的社区对于自己的成长是很重要的。

  1. 避免从特别成熟的项目开始。特别成熟的项目往往有许多贡献者，很难抢到一个入门级issue来开始第一个PR；且如果不能自己独立解决一个PR的话，可能这个PR会直接关闭。由于缺乏渐进式的引导，会缺少良好的参与感。

  2. 避免从特别小的项目开始。小项目发展不完善，很有可能存在缺少相关的贡献文档/没有代码规范/issue少等等问题。

  3. 选择知名开源软件基金会的孵化项目。这些项目不是特别成熟，对新贡献者友好；又不会特别不成熟，给人很差的参与体验。下面是几个推荐链接：

    1. CNCF沙箱项目
    2. CNCF孵化项目（包括毕业项目）
    3. Apache孵化项目

  4. 也可以直接选择从 CNCF 沙箱项目 DevStream 或者 Apache 孵化项目 Apache DevLake，以此敲开开源世界的大门。

二、寻找贡献点

你能对开源项目做很多方面的贡献，包括：

  • 开发新特性（feature）
  • 修复bug
  • 完善文档
  • 提出issue
  • 完善测试用例
  • …

在项目的Issue部分，一般可以通过筛选标签good first issue来找到适合新手入门的issue。如果有感兴趣的issue并且这个issue并没有被分配的话，可以在下面留言，等待项目管理者分配任务后就可以着手解决了。

三、如何提交PR？

一般开源项目的根目录都会有类似 CONTRIBUTION.md 或者类似名字的文档来介绍如何开始贡献。

3.1 第一步：Fork项目

首先需要在项目页面点击fork按钮将项目复制到自己的账号下。

在自己的repository中找到复制的项目，接着就可以开始修改了。将变更提交到自己fork的项目中，再通过pull request的方式将commit合入u上游项目。

3.2 第二步：在本地克隆项目

通过Git实现。

克隆项目的指令可以通过点击项目右上角的Code，再找到SSH链接，点击复制得到。

可以通过git remote -v查看对应的项目信息。

本地代码的变更永远只提交到 origin ，再通过 pull request 提交到上游项目。

3.3 第三步：更新本地代码分支（重要！）

每一次开始准备修改代码前，都需要确认本地的代码是最新的。如果基于老代码更新，很有可能遭遇到冲突等问题。

  • 更新本地 main 分支代码：

  • 不建议直接在 main 分支更新代码，避免同时修改多个部分代码需要提交多个PR而导致混乱的情况。最好通过创建一个新的分支，例如 feat-xxx 或者 fix-xxx 等更多可读性强的分支来开始工作。

  • 创建并切换到新分支： git checkout -b new-branch

3.6 第六步：开一个PR

在完成push操作后，GitHub的页面上会出现一个黄色提醒框，提示可以开一个 pull request。

填写标题（通常与 commit message 保持一致），然后按模板填写正文的内容。PR的模板可能每个项目都不尽相同，需要仔细查看并填写。

提交了PR后，就可以在PR list里找到自己的PR了，这时候还需要注意 CI 检查是否能全部通过，如果失败了还需要及时修复。

3.7 当PR遇到了问题

多数情况下，提交一个PR不会被马上并入，reviewer可能会提出很多修改意见，需要更新代码。

一般我们遵守一个约定：在 review 开始之前，更新代码尽量不引入新的 commits 记录，也就是能合并就合并，保证 commits 记录清晰且有意义；在 review 开始之后，针对 reviewers 的修改意见所产生的新 commit，可以不向前合并，这样能够让二次 review 工作更有针对性。

不过不同社区要求不一样，可能有的开源项目会要求一个 PR 里只能包含一个 commit，大家根据实际场景灵活判断即可。

3.8 如何合并commits？

3.8.1 通过Git命令行合并commits

有时候有些commit没有保留的意义，比如当提交完第一个commit之后发现代码少了一个单词或者缺少标点等等，这时候提交的第二个commit其实就可以和第一个合并，增加commit的可读性。

此时可以使用rebase命令来合并两个commit：git rebase -i HEAD~2。

完成合并后，需要通过git push -f origin feat-xxx强制更新。因为合并了commit本质上也是一种冲突，因此需要通过-f强制推送

3.8.2 图形化界面合并commit

  1. 点击右下角的 Git
  2. 选择想要合并的 commits
  3. 右键，然后点击 Squash Commits

3.9 PR产生了冲突如何解决？

即使每次写代码之前都更新代码，也不能完全避免出现冲突。当你的代码几天仍未并入，但是别人的代码可能修改了同一行代码还抢先并入了，此时就会产生冲突。

可以选择在线解决冲突或者本地解决冲突。
